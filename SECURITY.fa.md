## گزارش آسیب‌پذیری

ما امنیت KTTC را جدی می‌گیریم. اگر فکر می‌کنید آسیب‌پذیری امنیتی پیدا کرده‌اید، لطفاً آن را همانطور که در زیر شرح داده شده به ما گزارش دهید.

### کجا گزارش کنیم

**لطفاً آسیب‌پذیری‌های امنیتی را از طریق issue های عمومی GitHub گزارش نکنید.**

در عوض، لطفاً آنها را از طریق ایمیل به: **security@kt.tc** گزارش دهید

### چه چیزی را شامل کنیم

لطفاً اطلاعات زیر را در گزارش خود بگنجانید:

* نوع مشکل (مثلاً سرریز بافر، تزریق SQL، اسکریپت‌نویسی بین سایتی و غیره)
* مسیرهای کامل فایل(های) منبع مربوط به بروز مشکل
* محل کد منبع آسیب‌دیده (تگ/شاخه/commit یا URL مستقیم)
* هر پیکربندی خاصی که برای بازتولید مشکل لازم است
* دستورالعمل‌های گام به گام برای بازتولید مشکل
* کد اثبات مفهوم یا بهره‌برداری (در صورت امکان)
* تأثیر مشکل، از جمله نحوه بهره‌برداری مهاجم از مشکل

این اطلاعات به ما کمک می‌کند تا گزارش شما را سریع‌تر ارزیابی کنیم.

### چه انتظاری داشته باشیم

* باید ظرف **48 ساعت** تأییدیه دریافت کنید
* ما ظرف **7 روز** پاسخ دقیق‌تری ارسال خواهیم کرد که مراحل بعدی را نشان می‌دهد
* ما شما را از پیشرفت به سوی رفع و اعلامیه کامل مطلع خواهیم کرد
* ممکن است اطلاعات یا راهنمایی بیشتری درخواست کنیم

### سیاست افشا

* مسائل امنیتی با بالاترین اولویت رسیدگی خواهند شد
* هدف ما رفع آسیب‌پذیری‌های حیاتی ظرف **30 روز** است
* افشای عمومی پس از انتشار وصله انجام خواهد شد
* ما در مشاوره امنیتی از شما تقدیر خواهیم کرد (اگر بخواهید)

## بهترین شیوه‌های امنیتی

هنگام استفاده از KTTC، لطفاً این بهترین شیوه‌های امنیتی را دنبال کنید:

### کلیدهای API

* **هرگز کلیدهای API را در کنترل نسخه commit نکنید**
* از متغیرهای محیطی برای همه کلیدهای API استفاده کنید:
  ```bash
  export OPENAI_API_KEY="your-key-here"
  export ANTHROPIC_API_KEY="your-key-here"
  ```
* از فایل‌های `.env` (که gitignore شده‌اند) برای توسعه محلی استفاده کنید
* کلیدها را به طور منظم چرخش دهید

### اعتبارسنجی ورودی

* همیشه ورودی کاربر را اعتبارسنجی و پاکسازی کنید
* هنگام پردازش ترجمه‌ها از منابع غیرقابل اعتماد محتاط باشید
* از اعتبارسنجی داخلی در مدل‌های TranslationTask استفاده کنید

### وابستگی‌ها

* وابستگی‌ها را به‌روز نگه دارید: `pip install -U ".[dev]"`
* امنیت وابستگی را بررسی کنید با: `pip-audit` (توصیه می‌شود)
* مشاوره‌های امنیتی در مورد وابستگی‌ها را زیر نظر داشته باشید

### استقرار تولید

* از HTTPS برای همه ارتباطات API استفاده کنید
* محدودیت نرخ را برای نقاط پایانی API پیاده‌سازی کنید
* رویدادهای مرتبط با امنیت را ثبت کنید
* با حداقل مجوزهای لازم اجرا کنید
* از محیط‌های مجازی برای جداسازی وابستگی‌ها استفاده کنید

## ملاحظات امنیتی شناخته‌شده

### فراخوانی‌های API مدل زبانی بزرگ

* همه داده‌های ترجمه به ارائه‌دهندگان LLM (OpenAI، Anthropic) ارسال می‌شود
* **از KTTC برای محتوای حساس/محرمانه استفاده نکنید** مگر اینکه توافقنامه‌های پردازش داده مناسب داشته باشید
* سیاست‌های مدیریت داده ارائه‌دهنده LLM خود را بررسی کنید

### تزریق پرامپت

* مراقب باشید که ورودی مخرب ممکن است تلاش کند حملات تزریق پرامپت انجام دهد
* سیستم شامل اقدامات حفاظتی اولیه است، اما همیشه خروجی‌های حیاتی را اعتبارسنجی کنید
* از ترجمه‌ها برای برنامه‌های حیاتی امنیتی بدون بررسی انسانی استفاده نکنید

### کش محلی

* ترجمه‌های کش‌شده به صورت محلی در SQLite ذخیره می‌شوند
* مجوزهای فایل مناسب را در پایگاه داده کش اطمینان حاصل کنید
* هنگام مدیریت داده‌های حساس، کش را پاک کنید
