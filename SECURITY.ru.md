# Политика безопасности

## Поддерживаемые версии

Мы выпускаем исправления для уязвимостей безопасности в следующих версиях:

| Версия  | Поддерживается     |
| ------- | ------------------ |
| 0.1.x   | :white_check_mark: |
| < 0.1   | :x:                |

## Сообщение об уязвимости

Мы серьезно относимся к безопасности KTTC. Если вы считаете, что обнаружили уязвимость в безопасности, пожалуйста, сообщите нам об этом, как описано ниже.

### Куда сообщать

**Пожалуйста, НЕ сообщайте об уязвимостях безопасности через публичные GitHub issues.**

Вместо этого, пожалуйста, сообщите о них по электронной почте: **security@kt.tc**

### Что включить

Пожалуйста, включите следующую информацию в ваш отчет:

* Тип проблемы (например, переполнение буфера, SQL-инъекция, межсайтовый скриптинг и т.д.)
* Полные пути к исходным файлам, связанным с проявлением проблемы
* Расположение затронутого исходного кода (тег/ветка/коммит или прямой URL)
* Любую специальную конфигурацию, необходимую для воспроизведения проблемы
* Пошаговые инструкции для воспроизведения проблемы
* Код подтверждения концепции или эксплойта (если возможно)
* Влияние проблемы, включая то, как злоумышленник может использовать эту проблему

Эта информация поможет нам быстрее обработать ваш отчет.

### Чего ожидать

* Вы должны получить подтверждение в течение **48 часов**
* Мы отправим более подробный ответ в течение **7 дней**, указывающий следующие шаги
* Мы будем держать вас в курсе прогресса в направлении исправления и полного объявления
* Мы можем попросить дополнительную информацию или руководство

### Политика раскрытия информации

* Проблемы безопасности будут обрабатываться с наивысшим приоритетом
* Мы стремимся исправить критические уязвимости в течение **30 дней**
* Публичное раскрытие будет сделано после выпуска исправления
* Мы укажем вас в благодарностях в рекомендации по безопасности (если вы пожелаете)

## Лучшие практики безопасности

При использовании KTTC, пожалуйста, следуйте этим лучшим практикам безопасности:

### API-ключи

* **Никогда не коммитьте API-ключи** в систему контроля версий
* Используйте переменные окружения для всех API-ключей:
  ```bash
  export OPENAI_API_KEY="your-key-here"
  export ANTHROPIC_API_KEY="your-key-here"
  ```
* Используйте файлы `.env` (которые находятся в gitignore) для локальной разработки
* Регулярно меняйте ключи

### Валидация входных данных

* Всегда валидируйте и очищайте пользовательский ввод
* Будьте осторожны при обработке переводов из недоверенных источников
* Используйте встроенную валидацию в моделях TranslationTask

### Зависимости

* Держите зависимости в актуальном состоянии: `pip install -U ".[dev]"`
* Проверяйте безопасность зависимостей с помощью: `pip-audit` (рекомендуется)
* Отслеживайте рекомендации по безопасности зависимостей

### Развертывание в продакшене

* Используйте HTTPS для всех API-коммуникаций
* Реализуйте ограничение скорости для API-эндпоинтов
* Логируйте события, связанные с безопасностью
* Запускайте с минимальными необходимыми разрешениями
* Используйте виртуальные окружения для изоляции зависимостей

## Известные соображения безопасности

### Вызовы LLM API

* Все данные переводов отправляются провайдерам LLM (OpenAI, Anthropic)
* **Не используйте KTTC для конфиденциального контента**, если у вас нет соответствующих соглашений об обработке данных
* Ознакомьтесь с политиками обработки данных вашего провайдера LLM

### Инъекция промптов

* Помните, что вредоносный ввод может попытаться выполнить атаки инъекции промптов
* Система включает базовые защитные меры, но всегда валидируйте критические выводы
* Не используйте переводы для критически важных для безопасности приложений без человеческой проверки

### Локальное кэширование

* Кэшированные переводы хранятся локально в SQLite
* Обеспечьте правильные разрешения файлов для базы данных кэша
* Очищайте кэш при работе с конфиденциальными данными
