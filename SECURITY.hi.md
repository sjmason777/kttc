## कमजोरी की रिपोर्ट करना

हम KTTC की सुरक्षा को गंभीरता से लेते हैं। यदि आपको लगता है कि आपको सुरक्षा कमजोरी मिली है, तो कृपया नीचे बताए अनुसार हमें इसकी रिपोर्ट करें।

### कहाँ रिपोर्ट करें

**कृपया सार्वजनिक GitHub इश्यू के माध्यम से सुरक्षा कमजोरियों की रिपोर्ट न करें।**

इसके बजाय, कृपया उन्हें ईमेल के माध्यम से रिपोर्ट करें: **security@kt.tc**

### क्या शामिल करें

कृपया अपनी रिपोर्ट में निम्नलिखित जानकारी शामिल करें:

* समस्या का प्रकार (जैसे बफर ओवरफ्लो, SQL इंजेक्शन, क्रॉस-साइट स्क्रिप्टिंग, आदि)
* समस्या की अभिव्यक्ति से संबंधित स्रोत फ़ाइल(ओं) के पूर्ण पथ
* प्रभावित स्रोत कोड का स्थान (टैग/ब्रांच/कमिट या प्रत्यक्ष URL)
* समस्या को पुनरुत्पन्न करने के लिए आवश्यक कोई विशेष कॉन्फ़िगरेशन
* समस्या को पुनरुत्पन्न करने के लिए चरण-दर-चरण निर्देश
* प्रूफ-ऑफ-कॉन्सेप्ट या एक्सप्लॉइट कोड (यदि संभव हो)
* समस्या का प्रभाव, जिसमें यह भी शामिल है कि कोई हमलावर समस्या का फायदा कैसे उठा सकता है

यह जानकारी हमें आपकी रिपोर्ट को अधिक तेज़ी से ट्राइएज करने में मदद करेगी।

### क्या उम्मीद करें

* आपको **48 घंटों** के भीतर एक स्वीकृति प्राप्त होनी चाहिए
* हम अगले चरणों को दर्शाते हुए **7 दिनों** के भीतर एक अधिक विस्तृत प्रतिक्रिया भेजेंगे
* हम आपको फिक्स और पूर्ण घोषणा की प्रगति के बारे में सूचित करते रहेंगे
* हम अतिरिक्त जानकारी या मार्गदर्शन के लिए कह सकते हैं

### प्रकटीकरण नीति

* सुरक्षा समस्याओं को उच्चतम प्राथमिकता के साथ संभाला जाएगा
* हमारा लक्ष्य **30 दिनों** के भीतर महत्वपूर्ण कमजोरियों को पैच करना है
* पैच जारी होने के बाद सार्वजनिक प्रकटीकरण किया जाएगा
* हम सुरक्षा सलाहकार में आपको क्रेडिट देंगे (यदि आप चाहें)

## सुरक्षा सर्वोत्तम प्रथाएं

KTTC का उपयोग करते समय, कृपया इन सुरक्षा सर्वोत्तम प्रथाओं का पालन करें:

### API कीज़

* **API कीज़ को कभी भी संस्करण नियंत्रण में कमिट न करें**
* सभी API कीज़ के लिए पर्यावरण चर का उपयोग करें:
  ```bash
  export OPENAI_API_KEY="your-key-here"
  export ANTHROPIC_API_KEY="your-key-here"
  ```
* स्थानीय विकास के लिए `.env` फ़ाइलों का उपयोग करें (जो gitignored हैं)
* कीज़ को नियमित रूप से रोटेट करें

### इनपुट सत्यापन

* हमेशा उपयोगकर्ता इनपुट को मान्य और सैनिटाइज़ करें
* अविश्वसनीय स्रोतों से अनुवादों को संसाधित करते समय सावधान रहें
* TranslationTask मॉडल में बिल्ट-इन सत्यापन का उपयोग करें

### निर्भरताएं

* निर्भरताओं को अप टू डेट रखें: `pip install -U ".[dev]"`
* निर्भरता सुरक्षा की समीक्षा करें: `pip-audit` (अनुशंसित)
* निर्भरताओं पर सुरक्षा सलाहकारों की निगरानी करें

### उत्पादन परिनियोजन

* सभी API संचार के लिए HTTPS का उपयोग करें
* API एंडपॉइंट्स के लिए दर सीमा लागू करें
* सुरक्षा-प्रासंगिक घटनाओं को लॉग करें
* न्यूनतम आवश्यक अनुमतियों के साथ चलाएं
* निर्भरताओं को अलग करने के लिए वर्चुअल एनवायरनमेंट का उपयोग करें

## ज्ञात सुरक्षा विचार

### LLM API कॉल

* सभी अनुवाद डेटा LLM प्रदाताओं (OpenAI, Anthropic) को भेजा जाता है
* **संवेदनशील/गोपनीय सामग्री के लिए KTTC का उपयोग न करें** जब तक आपके पास उचित डेटा प्रसंस्करण समझौते न हों
* अपने LLM प्रदाता की डेटा हैंडलिंग नीतियों की समीक्षा करें

### प्रॉम्प्ट इंजेक्शन

* सावधान रहें कि दुर्भावनापूर्ण इनपुट प्रॉम्प्ट इंजेक्शन हमलों का प्रयास कर सकता है
* सिस्टम में बुनियादी सुरक्षा उपाय शामिल हैं, लेकिन हमेशा महत्वपूर्ण आउटपुट को मान्य करें
* मानव समीक्षा के बिना सुरक्षा-महत्वपूर्ण अनुप्रयोगों के लिए अनुवादों का उपयोग न करें

### स्थानीय कैशिंग

* कैश किए गए अनुवाद स्थानीय रूप से SQLite में संग्रहीत होते हैं
* कैश डेटाबेस पर उचित फ़ाइल अनुमतियां सुनिश्चित करें
* संवेदनशील डेटा को संभालते समय कैश साफ़ करें
